# Production Agent Studio Canon

## Non-Negotiable Architecture Principles

1. Product objective first, model objective second.
2. Product app and planning workspace are separate systems.
3. Postgres + pgvector is the durable product state plane.
4. Obsidian is the planning and design memory plane.
5. Realtime audio is a transport and turn-taking layer, not the whole reasoning system.
6. OpenRouter `deepseek/deepseek-v4-flash` is the default live-dialogue reasoning lane; Hugging Face/Gemma/Gamma/MLX are legacy, source-background, or optional non-default lanes unless a route explicitly selects them.
7. Every run is resumable, replayable, and inspectable through events and checkpoints.
8. Every major claim maps to a source record or is marked unsupported.
9. Retrieval quality is measured before generation quality is trusted.
10. Feedback becomes typed product data before it becomes memory.
11. Reliability is a release artifact: SLOs, error budgets, degradation policies, incident loops, and workflow reproducibility are required before production promotion.
12. Governance is route design: stakeholder requirements, non-accuracy objectives, slice evidence, business value, and reviewer capacity must be explicit before autonomy or optimization increases.
13. Kanban tracking is a vault planning artifact only; it must not be rendered as product cockpit or creator workflow UI.
14. Feedback-loop ledgers must distinguish accepted, revised, held, and rejected outcomes before autonomy treats a feedback gate as closed; pending, blocked, failed, or canceled linked work keeps the outcome held.
15. External publishing readiness must separate content policy review from local credential presence, normalize known platform aliases before lookup, block unsupported platform ids, and expose only env/file names or configured booleans, never secret values.

## Data Architecture Commitments

- Use append-only `run_events` as the audit trail.
- Give every durable event family an event-topic contract: partition key, ordering scope, retention/compaction policy, schema version, replay eligibility, and consumer groups.
- Promote event families only after schema compatibility, consumer cursor tracking, delivery semantics, idempotent sink behavior, projection rebuild evidence, partition-key hot-spot review, and audit-versus-compaction boundaries are explicit.
- Treat schema and API evolution as release-managed infrastructure: every event family, route API, tool schema, agent-card contract, MCP/A2A envelope, checkpoint state, source-ledger record, and provider adapter needs compatibility policy, consumer list, decoder status, deprecation window, and migration/test evidence before incompatible changes ship.
- Promote contract changes only after a compatibility release gate proves declared compatibility mode, served/storage versions, registered consumers, historical fixtures, decoder behavior, migration/backfill status, deprecation notices, and dual-read or dual-write bridges where needed.
- Treat MCP servers as governed connector releases, not generic plugin lists. A route may expose MCP only after protocol version, lifecycle/capability snapshot, schema dialect, stdio or Streamable HTTP transport, session/resume/cancel behavior, authorization discovery, resource/tool/prompt inventory, list-change drift handling, human approval policy, output validation, timeout/retry policy, retention/privacy review, and rollback are explicit.
- Promote ChatGPT app routes only after an app release gate proves MCP server identity, app record, tool descriptor versions, output schemas, widget templates, CSP/network review, structured-content versus `_meta` boundary tests, bridge-event coverage, state-scope policy, file authorization policy, auth grants/scopes, discovery evals, handler tests, UX/mobile screenshots, privacy/retention review, dependency review, submission/review status, fallback behavior, and rollback target.
- Treat specialist handoffs as A2A-like task boundaries rather than nested chat. Remote or local specialist routes need Agent Card identity, card freshness/hash, selected skill, task/context ID, typed artifact parts, stream and push-notification policy, cancellation semantics, auth boundary, idempotency, resume eligibility, terminal-state handling, and fallback before they can be promoted.
- Use `run_checkpoints` for durable state-machine resume and time-travel debugging.
- Treat memory and checkpointing as separate release-managed data planes. A route may write durable memory or claim resumability only after checkpoint persistence mode, pending-write recovery, replay/fork policy, memory scope, write approval, conflict policy, source evidence, consolidation policy, interrupt handling, trace redaction, retention, and rollback are explicit.
- Use workflow execution records for long-running route state and task queue contracts for background dispatch, backpressure, retries, leases, and quarantine.
- Treat deployed agent execution as a runtime plane with distinct graph blueprint, assistant config/version, thread state, run record, task queue, persistence backend, worker topology, stream subscription, interrupt/time-travel policy, scheduled-job policy, protocol endpoints, webhooks, double-texting behavior, scale/TTL policy, and rollback.
- Keep platform layers explicit: source/storage, compute, development environment, workflow scheduling, service orchestration, registries, evals, serving, and UI projections have separate owners, metadata, and failure surfaces.
- Define transaction consistency policies for route releases, source-state transitions, event projection updates, and side-effect outbox writes.
- Promote routes that mutate critical Postgres state only after a durable-state release gate proves transaction policies, WAL/PITR posture, restore drills, replica freshness, CDC/outbox duplicate handling, invariant checks, and schema migration safety.
- Define consistency contracts for every durable data family: required guarantee, ordering scope, stale-read tolerance, conflict policy, and delay-sensitivity tradeoff.
- Promote distributed data routes only after a distributed data contract release gate proves fault and synchrony assumptions, network partition policy, retry surfaces, consistency contracts, stale-state UX, replication topology, quorum caveats, linearizability boundaries, event partition/replay scope, notification-versus-persistent-event policy, conflict resolution, cross-entity transaction or saga boundaries, idempotency, recovery drills, fallback, and rollback.
- Use [[03-patterns/system-design/distributed-data-contracts]] as the reusable route-review checklist for distributed state. It separates persistent state-transition ledgers from realtime/UI notifications, object-level event ordering from tenant isolation, and cross-entity transaction/saga boundaries from hidden background partial completion.
- Use `sources`, `claims`, `retrieval_candidates`, and `retrieval_quality_ledgers` to control false positives and false negatives.
- Record extraction transforms, parser profiles, page coverage, layout elements, quality signals, filter decisions, source mixture experiments, and post-training data provenance before using any corpus for retrieval, eval generation, tuning, or route promotion.
- Promote document-derived retrieval, eval, or canon evidence only after a document-ingestion release gate proves parser profile/version, page coverage, layout-element lineage, table/figure quality, OCR/layout quality signals, layout-aware chunk policy, fallback decisions, privacy/provider boundary, known losses, and review decision.
- Treat web-scale filtering as an evaluated pipeline, not a cleanup script: sample inspection, format projection, extraction, language detection, PII masking, safety filtering, quality rules/classifiers, exact/near dedupe, tokenization/embedding, validation leakage checks, and runtime counters must be recorded before source data can support retrieval, eval generation, or tuning.
- Promote source-filter pipeline changes only after a source-filter pipeline release gate proves source projections, accepted and rejected sample inspections, filter versions and thresholds, false-positive/false-negative caveats, exact and near-dedupe provenance, stage counters, validation leakage checks, tokenization and embedding artifact versions, runtime benchmarks, and rollback snapshot.
- Treat reusable source, retrieval, route, feedback, and eval signals as feature-store-style definitions with entity keys, schemas, transformations, event time, offline history, online serving state, freshness, monitoring, consumers, and point-in-time leakage checks.
- Promote feature-consuming routes only after a feature parity release gate proves shared feature definitions, entity keys, transformation versions, offline and online store bindings, materialization jobs, point-in-time joins, leakage checks, freshness and drift monitors, stale/missing fallback behavior, authorized discovery, affected consumers, migration/deprecation handling, and sampled offline/online parity evidence.
- Treat agent work-surface automation as a platform lifecycle, not an assistant convenience. Source pipelines, feature/signal parity, model/provider records, artifact registries, deployment aliases, sampled outputs, delayed outcomes, MCP capability snapshots, skill instruction versions, render actions, explainability artifacts, and governance adoption burndown must stay linked.
- Promote work-surface automation routes only after a platform lifecycle/work-surface gate proves offline/online lifecycle records, parity checks, output logs, delayed outcome joins, MCP read/write scopes, local execution and credential boundaries, skill schemas/reference docs, render validation, rollback, responsible-AI review, and legacy route adoption state.
- Treat data-quality assertions as release-managed contracts: source snapshots, extracted artifacts, chunks, embeddings, indexes, features, eval datasets, route releases, and publishable artifacts need validation runs, failure samples, anomaly reports, actions, and exception waivers before promotion.
- Promote data-bearing assets only after a data-quality release gate links contract versions, validation runs/results, failure samples, anomaly reports, quality actions, exception waivers, affected consumers, and owner decisions.
- Treat model cards, source/dataset datasheets, route cards, and risk cards as release evidence backed by durable records. A Markdown card is a human-readable projection; the source of truth is the linked source, model, route, eval, risk, incident, and review records.
- Promote card-backed sources, models/providers, routes, generated-media lanes, publishing lanes, and high-impact eval suites only after a card projection release gate proves intended and unsupported use, stakeholder and affected-population context, source provenance and rights, dataset composition and exclusions, model/provider version and data-boundary review, slice and subgroup evals, content provenance, human-approval/security/privacy controls, supplier dependencies, monitoring, incident feedback, review cadence, and owner decision.
- Promote finetuning, adapters, prompt/prefix tuning, distillation, self-hosting, or model-objective changes only after a model-adaptation lifecycle release gate proves observed behavior evidence, mechanism assumptions, dataset documentation, rights policy, contamination checks, tokenizer and tokenization impact, objective family, selective-computation traces where used, rejected simpler interventions, capacity estimate, lifecycle cost, serving impact, fallback, and rollback.
- Treat Hugging Face Hub model, dataset, and benchmark metadata as registry input, not production proof. Hub repo revision, card metadata, license, gated/private access, scanner evidence, eval-result provenance, and leaderboard snapshots must be locally captured before a route uses a Hub model or dataset.
- Promote Hugging Face-sourced model, dataset, benchmark, or Space dependencies only after an `hf_hub_registry_release_gate` proves repo revision and artifact hashes, card snapshots, license/custom-license review, base-model and dataset lineage, gated/private access evidence, token/resource-group posture, malware/pickle/secrets/signed-commit/MFA evidence, Data Studio partial-view caveats, external eval and leaderboard provenance, route-specific evals, intended/unsupported-use review, redistribution/retention policy, fallback, and rollback.
- Treat privacy as route data processing, not a generic security note: every route that touches private/local/user/provider data needs processing activity, sensitive-data profile, de-identification, retention, provider boundary, deletion/export, and human-access audit records.
- Promote privacy-sensitive route changes only after a privacy release gate proves processing activity, sensitive-data profiles, de-identification/redaction policy and events, provider boundary, local/provider retention, deletion/export affordance, human-access audit, and third-party sharing caveats.
- Promote collaborative, personalized, social-facing, or high-agency routes only after a human-centered route release gate proves autonomy mode, user agency controls, collaboration process metrics, preference-perspective disagreement handling, personalization boundaries, privacy-norm evals, privacy attack simulations, anthropomorphism risk review, handoff visibility, memory-write policy, external-action policy, fallback, and rollback.
- Treat multi-agent topology as a release decision, not agent naming. Promote multi-agent, debate, critique, ranking, evolution, planner/checker, or expert-collaboration routes only after a collaborative-agent topology release gate proves task decomposition, selected and rejected topology alternatives, coordination contract, expected value, communication/tool overhead, latency/cost budget, reviewer-fatigue budget, error-propagation risk, role contracts, candidate-pool retention, loop-stop policy, domain boundary, scenario evals, human expert gate, fallback, rollback, and incident feedback.
- Treat adversarial evaluation, debate, red-team simulations, prompt-injection tests, and self-play-style improvement as strategic route surfaces. Promote them only after an adversarial route release gate proves players/agents, legal actions, utility/payoff contract, information-state assumptions, worst-case search traces, pruning rationale, cutoff-evaluator caveats, stochastic/chance-node handling, rollout policy, search budget, metareasoning stop conditions, human authority boundary, fallback, and rollback.
- Treat route scheduling and resource assignment as constraint satisfaction when hard feasibility matters. Promote constraint-aware planning only after a constraint release gate proves variables, domains, hard constraints, soft preferences, constraint graph/decomposition, propagation events, conflict sets, learned no-goods, repair behavior, human override, fallback, and rollback.
- Treat planning as a release-managed action-schema system, not a prose outline. Promote automated planning only after a planning release gate proves state facts, action preconditions/effects, planner-mode choice, heuristic relaxations, HTN refinements, contingent branches, execution monitoring, temporal/resource schedules, fallback, and rollback.
- Treat multiagent decision mechanisms as separate release decisions. A reviewer panel, specialist auction, contract-net delegation, coalition team, bargaining loop, or vote is not justified by "more agents"; it needs relationship model, payoff alignment, joint-action/resource constraints, truthful-reporting or manipulation caveat, aggregation rule, minority-safety evidence, human authority boundary, fallback, and rollback.
- Use `artifacts` and `artifact_revisions` for provenance and comparison.
- Treat object storage as the binary payload plane, not the metadata source of truth. Every binary artifact should bind a ledger record to bucket/container, key/name, immutable version or generation, content hash, MIME type, size, encryption posture, retention/lifecycle policy, and access grants.
- Treat lifecycle, retention, immutability, and signed-object access as route behavior. Tier movement, expiration, delete markers, restore latency, Object Lock/immutability, signed URLs, and SAS tokens can change replayability, privacy, and publication safety, so they need durable events and review policy.
- Promote binary evidence only after an object-artifact release gate proves immutable object version/generation, hash, storage policy, lifecycle/retention posture, signed-access history, restore drill evidence, and dependent route/eval/publication refs.
- Treat provider image/video generation as an artifact subsystem: media requests, parameter snapshots, async operations, retrieved output assets, safety filter results, variant sets, QA gates, provenance/disclosure evidence, and human approval are required before public use.
- Promote provider media-generation routes only after a provider-media release gate proves provider/model and endpoint currency, request schema version, parameter snapshots, input rights/consent, privacy class, async operation lifecycle, polling/webhook policy, retrieval events, output asset hashes, safety filter results, variant selection, visual/text-rendering evals, provenance/disclosure handoff, platform constraints, human approval, deprecation review, fallback route, and rollback target.
- Promote media embedding, self-supervised visual representation, cross-modal retrieval, visual duplicate, style-match, edit-reference, or storyboard-continuity routes only after a media-embedding release gate proves objective/checkpoint/transform provenance, source snapshot, rights/privacy filters, hard-negative coverage, robustness slices, task-surface evals, accepted/rejected candidate traces, refresh lineage, stale-reference handling, fallback, and rollback.
- Promote detection, segmentation, crop, edit-targeting, region-sensitive moderation, visual approval, or publish decisions only after a region-evidence release gate proves model/version/use scope, candidate sets, boxes, masks, frame refs, confidence and threshold policy, matching/suppression and postprocess policy, visual explanations, failure slices, edit/provenance refs, human review, fallback, rollback, and incident feedback.
- Promote video routes that claim action, temporal continuity, pacing, transition quality, subtitle alignment, audio/visual agreement, generated-video quality, or publishing readiness only after a temporal-video release gate proves clip-window and sampling policy, action/event records, object tracks, modality traces, disagreement policy, temporal eval cases, fallback route, human review, provenance, rollback, and incident feedback.
- Realtime voice audio artifacts need dual handling: local bounded artifact refs and hashes for audit/replay, but inline provider payloads for model calls when the provider cannot dereference local artifact URIs. Durable events and materialized turn records should carry metadata, not raw PCM, and local artifact capture needs explicit retention/cleanup settings before it is used in long-running sessions. Destructive local cleanup must be operator-visible so evidence-preservation runs can raise retention.
- Promote realtime voice stacks only after a realtime voice stack release gate proves session type, connection type, live-dialogue model, realtime/transport/ASR/TTS providers, OpenRouter route profile, LiveKit control-channel boundaries, turn policy, transcript/audio delta lineage, first transcript/audio timings, endpointing, interruption, cancellation, reconnect/recovery, playback state, voice rights/consent, audio format/alignment, text normalization, privacy/retention, latency SLO, quality evals, fallback, and rollback.
- Promote speech, dialogue-state, and extraction-backed graph routes only after a speech/dialogue/IE release gate proves provider/API version, credential boundary, audio configuration, turn/interruption policy, transcript-finalization policy, ASR/TTS eval slices, first-transcript/first-audio timings, realtime tool/MCP exposure, dialogue-state policy, extraction pipeline version, offset encoding, mention/referentiality/coreference candidate policy, entity candidate/NIL/canonicalization policy, graph-write review policy, privacy/retention posture, bias slice checks, fallback mode, and rollback target.
- Put reusable/deployable assets in registries with immutable versions and lineage: prompts, tools, agent graphs, model/adapters, source snapshots, datasets, extraction outputs, chunking profiles, vector/lexical/graph indexes, eval suites, media artifacts, and route releases.
- Promote reusable/deployable assets only after a registry-lineage release gate proves immutable versions, producer runs, input/output lineage events, validation state, rollback eligibility, and alias reassignment audit.
- Make binary evidence replayable: source PDFs, JPEGs, DOC/DOCX conversions, generated media, sandbox exports, batch manifests, eval outputs, trace attachments, and note artifacts can live in object storage, but Agent Studio should never rely on a mutable object key alone. Immutable version or generation IDs, hashes, lifecycle/retention rules, signed-access events, restore posture, and delete/tombstone events belong in the ledger.

- Promote vector database routes only after a vector-store release gate proves collection schema, source snapshot, vector fields, payload schema/indexes, required filter policy, hybrid query plan, multivector profile where applicable, quantization profile, alias/rollback pointer, snapshot/restore evidence, optimizer policy, partition policy, recall and latency/cost evals, and rollback target.

- Attach latency class and fallback policy to every provider call.
- Measure end-to-end latency and provider-local latency separately.
- Keep realtime interruption paths short and expert synthesis paths durable.
- Use provider stubs for local rehearsal, but mark them as non-production evidence.
- Product rehearsal paths may exercise realtime session, turn routing, spoken-response planning, and A2A handoff contracts without external credentials, but the UI and ledger must state that this does not prove LiveKit media, OpenRouter live-dialogue reasoning, Kokoro speech output, or provider-backed latency.
- Provider-backed voice smoke should use captured voice artifacts before synthetic probes. A smoke pass that can read a hashed local user-turn PCM artifact has stronger relevance for OpenRouter/Kokoro first-response latency than a synthetic silence input. Long-running runs need newest-first artifact selection, hash verification, safe path resolution, and race-safe file reads before the smoke result can be trusted.
- Product voice proof should render the captured-artifact decision directly. Operators should not have to inspect provider-smoke JSON to know whether the OpenRouter/Kokoro route used a persisted microphone artifact, which artifact/hash/turn was used, or whether the run fell back to the synthetic probe fixture.
- Product voice timing proof should render durable stage evidence directly. Operators should not have to inspect timing-ledger JSON to know whether LiveKit readiness, speech start, turn commit, OpenRouter reasoning start, first text/audio, and barge-in cancellation are measured, partial, or missing.
- A provider-backed voice-to-voice start cannot stop at browser room join. It must prove or start the backend OpenRouter/LiveKit/Kokoro participant, then require durable participant/presence evidence before claiming live voice readiness.
- Local development transport is part of the product proof loop. A local-first voice stack should expose LiveKit dev-server supervision as a separate transport control, because a running OpenRouter/Kokoro participant is not useful if the LiveKit media server is absent or unreachable.
- Local provider-backed voice bootstrap should make transport setup explicit: local LiveKit dev mode can be native or Compose-backed, `OPENROUTER_LIVEKIT_URL=ws://127.0.0.1:7880` is the local dev transport URL, and the local dev key/secret is not production security evidence. Runtime preflight should verify LiveKit server API connectivity, not only the presence of URL/key/secret strings; the connectivity timeout must be bounded, path-prefix proxy URLs must be explicit, and query/fragment URLs should fail before network calls.
- Kokoro proof should not overfit to hosted inference. The default spoken-output route is Kokoro, and first-audio smoke may prove Kokoro through either a hosted endpoint or a local Kokoro package. Evidence must name the chosen `kokoro_provider` and `kokoro_transport` so local open-weight TTS runs are not mislabeled as missing provider proof.
- OpenRouter endpoint proof must not overfit to credential presence. When runtime preflight is explicitly requested, OpenRouter readiness needs a production-path text-turn request through `deepseek/deepseek-v4-flash`, redacted provider failures, and a positive `openrouter-live-dialogue-reasoning` check; passive readiness must stay non-networked.
- Product preflight dispatch should use named route/preflight options, not positional boolean tuples. Full runtime preflight must explicitly include edge, agent, LiveKit, OpenRouter live dialogue, and Kokoro TTS probes; passive refreshes must remain a separate no-provider-call path. These option contracts should live near the runtime-readiness guard logic and be directly testable as values, so provider-call boundaries are not protected only by component-source formatting checks.
- Product proof language must not over-claim failed setup. A synthetic-probe fallback label is valid only after the OpenRouter/Kokoro smoke step actually ran and used probe audio; blocked or failed smoke needs explicit blocked/failed copy with the blocker or error surfaced to the creator/operator.
- Session-bound voice smoke is stronger than endpoint smoke. If the creator app has an active LiveKit session, live OpenRouter/Kokoro smoke should reuse that session id, require fresh participant proof for the same session, and block before external model calls when participant presence is missing or stale.
- Session-bound smoke must bind both control evidence and runtime inputs. It is insufficient to report a session id in the ledger while executing a synthetic runtime context; the turn sent to OpenRouter/Kokoro must carry the same session/room/participant context or the proof must be labeled endpoint-only.
- Active-room voice smoke must bind captured audio too. If a smoke run claims session-bound LiveKit proof, OpenRouter/Kokoro should receive a persisted user turn artifact from that same session; synthetic probe input is endpoint-only evidence and should block or be relabeled before active-room claims.
- Product live-smoke controls should make active-room proof fresh before spending provider calls. Send a new room-level presence probe, wait briefly for durable agent-ready evidence scoped to that probe/session, then run the backend smoke ledger; if same-session captured audio is still missing, block with a speak-first action instead of falling back to synthetic audio.
- Product voice UI state is a run/session-owned projection. Late presence probes, timing ledgers, smoke ledgers, and LiveKit data-channel events must be discarded after Stop, run replacement, or session replacement rather than repainting a stale active view.
- Voice-to-voice UX should expose the runtime state machine, not just a generic ready/offline badge. Product UI must map transport and agent events into disconnected, connecting, connected, listening, thinking, speaking, interrupting, reconnecting, ended, and failed, plus show room, mic, participant, and context-pruning signals.
- Interruption UX must distinguish request from acknowledgement. A frontend interrupt click is not proof that speech stopped; product proof should advance only when Rust edge acknowledgement or OpenRouter/Kokoro cancellation events confirm reasoning cancel, Kokoro buffer clear, and LiveKit output stop.
- Compatibility fallbacks for old artifacts must remain evidence-based. If explicit session metadata is missing, an artifact path may prove session binding only when it contains the requested session id; otherwise active-room smoke should block rather than infer ownership.
- Path-based compatibility checks must use path-segment equality, not substring matching, for identifiers such as realtime session ids.
- Hot event-family lookups need matching indexes. Any proof gate that queries a specific run/event-type family on always-on runs should have a `(run_id, event_type, event_id)` style access path, not only a generic run event cursor.
- Provider-smoke UI contracts should type known status values and treat unknown future statuses as unproven, not as successful local proof. Transport proof should expose both latency and implementation path, because `local_package` Kokoro and hosted HF Kokoro have different operator implications even when both satisfy the open-weight TTS requirement.
- Backend provider-smoke contracts should constrain emitted ledger statuses before they reach durable artifacts or product UI. Use explicit status enums for step/run results and keep JSON serialization string-compatible so API stability and proof correctness move together.
- Provider route decisions should be first-class evidence, not inferred in separate surfaces. For Kokoro, use one route object that can report `hf_endpoint`, `local_package`, or `missing`; readiness, smoke blockers, live smoke proof, and UI summaries should all consume that route so the user sees the same setup truth everywhere.

- Every route chooses among function, agent, workflow, multi-agent team, and external A2A route explicitly. If deterministic code can handle the task, that remains the default.
- Promote Agent Framework-style runtime routes only after a release gate proves framework version, component choice, rejected deterministic alternative, agent/workflow/team/A2A mode, model/provider binding, credential selection, session/history/context/audit provider policy, typed workflow executor and edge contracts, HITL wait/resume behavior, checkpoint policy, A2A card/capability/context mapping, streaming/background continuation policy, middleware/tool/MCP bindings, team pattern and stop policy, team edit policy, observability, benchmark/eval evidence, third-party boundary review, responsible-AI mitigations, fallback, and rollback.

## Product UX Commitments

- The app feels like a live conversational studio.
- The app includes voice/text conversation, draft previews, source ledger, artifact browser, feedback controls, voice settings, and run activity.
- Run activity must make failed, blocked, and canceled specialist work visible as attention items, sorted by latest update, with redacted summaries and a guarded requeue-and-run action only when execution is message-targeted. Retry authorization alone is not enough: the backend must receive the exact retried message id, validate run/recipient/status, avoid unrelated inbox mutation, and refuse default-roster fallback for unresolved target ids.
- Live voice follow-up execution must start from the exact persisted follow-up message and continue only through accepted child-message lineage. Do not wake a broad worker inbox after an assistant voice event; older accepted realtime or recovery tasks must remain untouched unless they are descendants of the current follow-up.
- The app exposes always-on/autonomous work as compact run controls backed by durable worker profiles and heartbeats, not as a project-management board.
- Product UI async work must carry explicit run ownership: route/compose, worker continuation, feedback, production actions, Autopilot controls, and realtime voice startup write state only when the active run id and run-version token still match.
- Run-ownership policy should be executable, not just documented: stale async completion tests must cover version mismatch, wrong-run busy preservation, no-run busy preservation, same-version duplicate no-run submits, and old completions landing after a newer run or clean no-run session takes over.
- Always-on profile UX must show durable execution proof, not just active/off state: bind heartbeat evidence to the selected autonomous profile, prefer persisted heartbeat ledger artifacts, and never borrow evidence from another profile.
- Creator-facing always-on controls may expose bounded execution actions such as `Heartbeat` when they directly refresh durable run evidence; they must stay execution/status oriented and avoid becoming planning-board controls.
- Product scheduler wakes must be scoped to the current run and expected execution mode before they are exposed in the creator app. Global scheduler sweeps remain operator/CLI behavior; the creator UI may only run due Autopilot profiles for the active run and should show scheduler proof only when the durable event names the active profile.
- Creator-facing always-on actions must be run-owned and profile-owned. Duplicate in-flight actions should surface visible state, not throw, and stale profile/run responses must be ignored before they can overwrite the current studio session.
- Always-on evidence should fail closed in the UI: if the current run has no Autopilot profile, old heartbeat artifacts or events are not enough to render Autopilot proof.
- Product Activity must read a latest-event tail, not a first-page event sample, when it renders live proof for recent autonomous work. Scheduler and heartbeat evidence are operational proof surfaces, so the newest worker events must be available without forcing users into an operator cockpit.
- The app excludes planning boards and planning architecture pages.
- Obsidian and HTML/JSON viewers are for our planning and design inspection.
- Shareable interactive HTML run notes must sanitize embedded durable state and title text before writing files or recording note artifacts; event payloads, artifact content/provenance, and provider exception strings are not safe merely because the surface is outside the product app.

## First Build Sequence

1. Stabilize provider config, Postgres schema, transaction consistency policies, and agent-card registry.
2. Finish deterministic state-machine routes for run creation, event logging, event-topic contracts, workflow execution records, checkpointing, feedback, artifact provenance, and source ledgers.
3. Build provider smoke tests for search, rerank, OpenRouter realtime dialogue, legacy Gemma/HF routes, realtime audio, and imagegen boundaries.
4. Add baseline SLO/error-budget/degradation records for the first run, retrieval, voice, artifact, and ingestion surfaces.
5. Add dependency timeout/retry/jitter/idempotency defaults, task queue contracts, DLQ/quarantine policies, autoscaling/admission policies, and separate concurrency pools for realtime, search/retrieval, model inference, MCP/browser tools, media generation, ingestion, and evals.
6. Build the first cockpit slice around conversation, drafts, sources, artifacts, and feedback gates.
7. Add PITR/restore drills and replica-freshness checks for the Postgres durable plane.
8. Add durable-state release gates for critical Postgres write paths before source-ledger, route-release, artifact, memory, publishing, feedback, billing, or eval-state mutation is considered production-ready.
9. Add replay drills for run-state, artifact-state, source-freshness, route-release, and eval-result projections.
10. Add schema/API compatibility tests, decoder compatibility checks, and deprecation/migration ledgers for event families, route APIs, provider adapters, and tool contracts.
11. Add registry and lineage records for prompts, model/provider routes, source snapshots, indexes, eval suites, generated artifacts, aliases, and route releases.
12. Do not promote provider routes from presence alone. Endpoint values need shape validation before readiness evidence, and malformed config must be recorded as a blocker or explicit fallback reason rather than hidden behind a generic provider route.
13. Make runtime constructors fail fast from the validated route contract. A route that is not ready should not defer its first clear error until provider call time.
14. Add retrieval quality ledgers and graph-backed evidence traversal before trusting source-backed generation.
15. Add live model routes only after provider stubs, event contracts, registry lineage, governance gates, and reliability gates pass.
16. Add provider readiness gates for API-key boundary, project budget, safety identifiers, rate-limit observations, quota-aware backoff, latency/cost strategy, and user-reporting loops before enabling external publishing or high-volume automation.
17. Add prompted-component and chain-definition release records so prompt, route graph, model, grounding snapshot, tools, evals, and safety configs cannot drift independently.
